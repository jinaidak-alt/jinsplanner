### ✅ C-3. "전체 데이터 초기화" 버튼이 로그인 상태에서 무효

### ✅ C-4. `calcNotifTime` UTC/로컬 혼용 — 새벽 알림 전날 발송

**현상**
- [index.html:2673-2680](index.html#L2673-L2680):
  ```js
  function calcNotifTime(dateStr, timeStart, offsetMin) {
    const notifDate = new Date(`${dateStr}T${timeStart}:00`);  // 로컬 시간 파싱
    notifDate.setMinutes(notifDate.getMinutes() - offsetMin);
    const h = String(notifDate.getHours()).padStart(2, '0');   // 로컬 시
    const m = String(notifDate.getMinutes()).padStart(2, '0'); // 로컬 분
    const d = notifDate.toISOString().slice(0, 10);            // ← UTC 날짜!
    return { date: d, time: `${h}:${m}` };
  }
  ```

**영향**
- KST(UTC+9) 기준 00:00 ~ 08:59 사이 알림 시각이면 `toISOString()`의 날짜가 하루 앞선 날짜(UTC)로 반환
- 예: 2026-04-24 08:00 KST → UTC는 2026-04-23 23:00 → `date="2026-04-23"`, `time="08:00"`
- Swift가 받는 payload: "2026-04-23 08:00에 알림 예약" → **실제로 원하는 날짜보다 하루 빨리 울림**
- 특히 `alarmOffset=30분 전` 설정 시 08:30 알림이 08:00으로 계산되면 전날 발송 가능

**수정 제안**
- `d`를 로컬 날짜 기준으로 생성:
  ```js
  const y = notifDate.getFullYear();
  const mo = String(notifDate.getMonth()+1).padStart(2,'0');
  const da = String(notifDate.getDate()).padStart(2,'0');
  const d = `${y}-${mo}-${da}`;
  ```
  또는 기존 `toStr()` 헬퍼 재사용.

### ✅ C-6. 반복 루틴 편집 시 `excludedDates`·`endDate` 유실

**현상**
- [index.html:1885-1897](index.html#L1885-L1897) `submitRoutine()` 내부에서 entry 객체를 **새로 구성**:
  ```js
  const entry = {
    id: editId||...,
    text, categoryId, time, timeSlot, repeat,
    startDate: ..., memo: ...,
  };
  ```
- `excludedDates`, `endDate` 필드가 entry에 포함되지 않음 → 기존 루틴 편집 시 덮어씌우면서 유실

**영향**
- "오늘만 수정"으로 제외해둔 날짜가 전체 수정 후 다시 루틴으로 노출됨

**수정 제안**
- entry 구성 시 기존 필드 보존:
  ```js
  const prev = editId ? routines.find(r=>r.id===editId) : null;
  const entry = { ..., excludedDates: prev?.excludedDates, endDate: prev?.endDate };
  ```

## 🟠 High

### ✅ H-1. Firestore → localStorage 로드 중 race condition (데이터 유실 위험)

**현상**
- [index.html:3148-3163](index.html#L3148-L3163): 로그인 감지 시 `getDoc()` await 중 사용자가 체크 버튼을 누르면
  1. `saveChecks()` → `saveToCloud()` 호출
  2. `saveToCloud`는 localStorage 값을 Firestore에 씀
  3. 아직 `getDoc()` 결과가 localStorage에 반영되기 전이므로 **이전 세션/다른 기기의 데이터가 덮어써질 수 있음**

**영향**
- 새 기기 첫 로그인 시 또는 오랜만에 접속했을 때, 다른 기기에서 쌓인 최신 데이터가 날아감
- 드물게 발생하지만 치명적 (복구 불가)

**수정 제안**
- `window._loadingCloud = true` 플래그를 `getDoc` 전에 세우고 완료 후 해제
- `saveToCloud()` 초입에 `if (window._loadingCloud) return` 가드
- 또는 `onAuthStateChanged` 핸들러를 먼저 완료시키고 UI 렌더 — 현재는 UI 먼저 렌더되고 이후 덮어씀

### ✅ H-3. 완료된 테스크 영구 누적 → Firestore 1MB 한계 위험

> ⚠️ **부분 처리**: `saveToCloud`에 try/catch + 사용자 알림만 추가됨. 자동 아카이빙/정리 정책은 미구현 — 장기 사용 시 여전히 누적됨. 필요 시 별도 이슈로 추적.

**현상**
- `tasks`는 완료되어도 자동 삭제되지 않음

**영향**

- 저장 실패 시 에러 핸들링 없음 → 사용자는 저장된 줄 알지만 실제 Firestore 쓰기 실패

**수정 제안**
- 완료된 테스크 N일 경과 시 자동 아카이브/삭제 옵션
- `saveToCloud` 에러 캐치 + 사용자 알림
- 장기적으로는 Firestore subcollection으로 분리 (`users/{uid}/tasks/{id}`)
